# 完成工作 - 提交前检查清单

在提交或提交之前，使用此检查清单确保工作完整性。

**时机**：代码编写和测试完成后，提交之前

---

## 检查清单

### 1. 代码质量

```bash
# 必须通过
pnpm lint
pnpm type-check
pnpm test
```

- [ ] `pnpm lint` 通过且无错误？
- [ ] `pnpm type-check` 通过且无类型错误？
- [ ] 测试通过？
- [ ] 没有 `console.log` 语句（使用 logger）？
- [ ] 没有非空断言（`x!` 操作符）？
- [ ] 没有 `any` 类型？

### 2. 文档同步

**结构文档**：
- [ ] `.trellis/spec/backend/` 需要更新吗？
  - 新模式、新模块、新约定
- [ ] `.trellis/spec/frontend/` 需要更新吗？
  - 新组件、新 hook、新模式
- [ ] `.trellis/spec/guides/` 需要更新吗？
  - 新的跨层流程、bug 经验

**关键问题**：
> "如果我修复了一个 bug 或发现了非显而易见的东西，我是否应该记录它，这样未来的我（或其他人）不会遇到同样的问题？"

如果是 → 更新相关规范文档。

### 3. API 变更

如果修改了 API 端点：

- [ ] 输入模式更新了？
- [ ] 输出模式更新了？
- [ ] API 文档更新了？
- [ ] 客户端代码更新以匹配？

### 4. 数据库变更

如果修改了数据库模式：

- [ ] 迁移文件创建了？
- [ ] 模式文件更新了？
- [ ] 相关查询更新了？
- [ ] 种子数据更新了（如适用）？

### 5. 跨层验证

如果变更跨越多层：

- [ ] 数据在各层正确流转？
- [ ] 各边界错误处理正常？
- [ ] 类型在各层一致？
- [ ] 加载状态处理了？

### 6. 手动测试

- [ ] 功能在浏览器/应用中正常？
- [ ] 边缘情况测试了？
- [ ] 错误状态测试了？
- [ ] 页面刷新后正常？

---

## 快速检查流程

```bash
# 1. 代码检查
pnpm lint && pnpm type-check

# 2. 查看变更
git status
git diff --name-only

# 3. 根据变更的文件，检查上述相关项
```

---

## 常见遗漏

| 遗漏 | 后果 | 检查 |
|-----------|-------------|-------|
| 结构文档未更新 | 其他人不知道变更 | 检查 .trellis/spec/ |
| 迁移未创建 | 模式不同步 | 检查 db/migrations/ |
| 类型未同步 | 运行时错误 | 检查共享类型 |
| 测试未更新 | 虚假信心 | 运行完整测试套件 |
| 留下 console.log | 生产日志噪音 | 搜索 console.log |

---

## 与其他命令的关系

```
开发流程：
  写代码 -> 测试 -> /trellis:finish-work -> git commit -> /trellis:record-session
                          |                              |
                   确保完整性              记录进度

调试流程：
  遇到 bug -> 修复 -> /trellis:break-loop -> 知识捕获
                       |
                  深度分析
```

- `/trellis:finish-work` - 检查工作完整性（本命令）
- `/trellis:record-session` - 记录会话和提交
- `/trellis:break-loop` - 调试后的深度分析

---

## 核心原则

> **交付不仅包括代码，还包括文档、验证和知识捕获。**

完成的工作 = 代码 + 文档 + 测试 + 验证
