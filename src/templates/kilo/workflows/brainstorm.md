# 头脑风暴 - 需求发现（AI 编码增强版）

引导 AI 在**实施之前**进行协作式需求发现，为 AI 编码工作流优化：

* **任务优先**（立即捕捉想法）
* **先行动后提问**（减少低价值问题）
* **研究优先**技术选择（避免让用户凭空构思方案）
* **发散 → 收敛**（扩展思维，然后锁定 MVP）

---

## 使用时机

当用户从 `/trellis:start` 描述开发任务时触发，特别适用于：

* 需求不清晰或正在演变
* 有多条有效的实现路径
* 权衡很重要（用户体验、可靠性、可维护性、成本、性能）
* 用户可能一开始不知道最佳选项

---

## 核心原则（不可协商）

1. **任务优先（尽早捕捉）**
   确保一开始就有任务存在，这样用户的想法会被立即记录下来。

2. **先行动后提问**
   如果你能从代码库、文档、配置、惯例或快速研究中推导答案——先去做。

3. **每条消息一个问题**
   不要用一系列问题淹没用户。问一个，更新 PRD，再重复。

4. **倾向具体选项**
   对于偏好/决策问题，提出 2-3 个可行且具体的方法，并说明权衡。

5. **技术选择研究优先**
   如果决策取决于行业惯例/类似工具/既定模式，先做研究，然后提出方案。

6. **发散 → 收敛**
   初步理解后，主动考虑未来的演进、相关场景和失败/边缘情况——然后收敛到一个有明确范围外的 MVP。

7. **不要问元问题**
   不要问"我应该搜索吗？"或"你能贴出代码让我继续吗？"
   如果需要信息：搜索/检查。如果受阻：提出最小的阻塞问题。

---

## 步骤 0：确保任务存在（始终执行）

在任何问答之前，确保任务存在。如果不存在，立即创建一个。

* 使用从用户消息派生的**临时工作标题**。
* 标题不完美也没关系——稍后在 PRD 中完善。

```bash
TASK_DIR=$(python3 ./.trellis/scripts/task.py create "brainstorm: <short goal>" --slug <auto>)
```

立即用已知内容创建/填充 `prd.md`：

```markdown
# brainstorm: <short goal>

## 目标

<一段话：什么 + 为什么>

## 我已经知道的

* <来自用户消息的事实>
* <从代码库/文档中发现的事实>

## 假设（临时的）

* <需要验证的假设>

## 待解决的问题

* <仅阻塞/偏好问题；保持列表简短>

## 需求（演变的）

* <从已知内容开始>

## 验收标准（演变的）

* [ ] <可测试的标准>

## 完成定义（团队质量门槛）

* 添加/更新测试（适当的地方进行单元/集成测试）
* Lint / 类型检查 / CI 绿色
* 如果行为变更则更新文档/笔记
* 如果有风险则考虑发布/回滚

## 范围之外（明确说明）

* <我们在这个任务中不会做的事情>

## 技术笔记

* <检查的文件、约束、链接、引用>
* <研究笔记摘要（如适用）>
```

---

## 步骤 1：自动上下文（在提问之前做这个）

在问"代码是什么样的？"之类的问题之前，自己收集上下文：

### 代码库检查清单

* 识别可能受影响的模块/文件
* 找到现有模式（类似功能、惯例、错误处理风格）
* 检查配置、脚本、现有命令定义
* 记录任何约束（运行时、依赖策略、构建工具）

### 文档检查清单

* 查找现有的 PRD/规范/模板
* 查找命令使用示例、README、ADR（如有）

将发现写入 PRD：

* 添加到"我已经知道的"
* 将约束/链接添加到"技术笔记"

---

## 步骤 2：分类复杂度（仍然有用，但不阻塞任务创建）

| 复杂度 | 标准 | 操作 |
| ------------ | ------------------------------------------------------ | ------------------------------------------- |
| **简单** | 单行修复、拼写错误、明显的更改 | 跳过头脑风暴，直接实施 |
| **简单** | 目标清晰，1-2 个文件，范围明确 | 问 1 个确认问题，然后实施 |
| **中等** | 多个文件，有些歧义 | 轻度头脑风暴（2-3 个高价值问题） |
| **复杂** | 目标模糊、架构选择、多种方法 | 完整头脑风暴 |

> 注意：任务从步骤 0 已经存在。分类只影响头脑风暴的深度。

---

## 步骤 3：问题门槛（只问高价值问题）

在问任何问题之前，运行以下门槛：

### 门槛 A — 我能否不通过用户推导出来？

如果答案可以通过以下方式获得：

* 代码库检查（代码/配置）
* 文档/规范/惯例
* 快速市场/开源研究

→ **不要问。** 获取它，总结，更新 PRD。

### 门槛 B — 这是元问题/懒问题吗？

示例：

* "我应该搜索吗？"
* "你能贴出代码让我继续吗？"
* "代码是什么样的？"（当代码库可用时）

→ **不要问。** 采取行动。

### 门槛 C — 这是什么类型的问题？

* **阻塞**：没有用户输入无法继续
* **偏好**：多个有效选择，取决于产品/用户体验/风险偏好
* **可推导**：应该通过检查/研究来回答

→ 只问**阻塞**或**偏好**问题。

---

## 步骤 4：研究优先模式（技术选择必须使用）

### 触发条件（任一 → 研究优先）

* 任务涉及选择方法、库、协议、框架、模板系统、插件机制或 CLI 用户体验约定
* 用户要求"最佳实践"、"别人怎么做"、"推荐"
* 用户无法合理列举选项

### 研究步骤

1. 识别 2-4 个可比较的工具/模式
2. 总结常见惯例及其存在原因
3. 将惯例映射到我们代码库的约束
4. 为我们的项目产生 **2-3 个可行方法**

### 研究输出格式（PRD）

在 PRD 中添加一个部分（在技术笔记内或独立部分）：

```markdown
## 研究笔记

### 类似工具的做法

* ...
* ...

### 我们代码库/项目的约束

* ...

### 这里可行的方法

**方法 A: <名称>**（推荐）

* 工作原理：
* 优点：
* 缺点：

**方法 B: <名称>**

* 工作原理：
* 优点：
* 缺点：

**方法 C: <名称>**（可选）

* ...
```

然后问**一个**偏好问题：

* "你更倾向哪种方法：A / B / C（或其他）？"

---

## 步骤 5：扩展扫描（发散）— 初步理解后必需

在你能够总结目标后，在收敛之前主动扩展思维。

### 扩展类别（每项保持 1-2 点）

1. **未来演进**

   * 这个功能在 1-3 个月内可能变成什么？
   * 哪些扩展点值得现在保留？

2. **相关场景**

   * 哪些相邻命令/流程应该与此保持一致？
   * 是否有对等期望（创建 vs 更新、导入 vs 导出等）？

3. **失败和边缘情况**

   * 冲突、离线/网络故障、重试、幂等性、兼容性、回滚
   * 输入验证、安全边界、权限检查

### 扩展消息模板（给用户）

```markdown
我理解你想实现：<当前目标>。

在深入设计之前，让我快速发散考虑三类（以避免以后返工）：

1. 未来演进：<1-2 点>
2. 相关场景：<1-2 点>
3. 失败/边缘情况：<1-2 点>

对于这个 MVP，你想包含哪些（或者都不）？

1. 仅当前需求（最小可行）
2. 添加 <X>（为未来扩展保留）
3. 添加 <Y>（提高健壮性/一致性）
4. 其他：描述你的偏好
```

然后更新 PRD：

* MVP 包含的内容 → `需求`
* 排除的内容 → `范围之外`

---

## 步骤 6：问答循环（收敛）

### 规则

* 每条消息一个问题
* 尽可能使用多项选择
* 用户每次回答后：

  * 立即更新 PRD
  * 将已回答的项目从`待解决问题` → `需求`
  * 用可测试的复选框更新`验收标准`
  * 澄清`范围之外`

### 问题优先级（推荐）

1. **MVP 范围边界**（包含/排除什么）
2. **偏好决策**（在提出具体选项后）
3. **失败/边缘行为**（仅针对 MVP 关键路径）
4. **成功指标和验收标准**（什么证明它有效）

### 首选问题格式（多项选择）

```markdown
对于 <topic>，你更倾向哪种方法？

1. **选项 A** — <含义 + 权衡>
2. **选项 B** — <含义 + 权衡>
3. **选项 C** — <含义 + 权衡>
4. **其他** — 描述你的偏好
```

---

## 步骤 7：提出方法并记录决策（复杂任务）

在需求足够清晰后，提出 2-3 个方法（如果还没有通过研究优先模式做）：

```markdown
根据当前信息，这里有 2-3 个可行的方法：

**方法 A: <名称>**（推荐）

* 如何做：
* 优点：
* 缺点：

**方法 B: <名称>**

* 如何做：
* 优点：
* 缺点：

你更倾向哪个方向？
```

在 PRD 中将结果记录为 ADR-lite 部分：

```markdown
## 决策（ADR-lite）

**背景**：为什么需要这个决策
**决策**：选择了哪种方法
**后果**：权衡、风险、潜在未来改进
```

---

## 步骤 8：最终确认 + 实施计划

当待解决的问题解决后，用结构化摘要确认完整需求：

### 最终确认格式

```markdown
以下是我对完整需求的理解：

**目标**：<一句话>

**需求**：

* ...
* ...

**验收标准**：

* [ ] ...
* [ ] ...

**完成定义**：

* ...

**范围之外**：

* ...

**技术方法**：
<简要总结 + 关键决策>

**实施计划（小 PR）**：

* PR1: <脚手架 + 测试 + 最小基础设施>
* PR2: <核心行为>
* PR3: <边缘情况 + 文档 + 清理>

这看起来正确吗？如果是的，我将开始实施。
```

### 子任务分解（复杂任务）

对于有多个独立工作项的复杂任务，创建子任务：

```bash
# 创建子任务
CHILD1=$(python3 ./.trellis/scripts/task.py create "Child task 1" --slug child1 --parent "$TASK_DIR")
CHILD2=$(python3 ./.trellis/scripts/task.py create "Child task 2" --slug child2 --parent "$TASK_DIR")

# 或链接现有任务
python3 ./.trellis/scripts/task.py add-subtask "$TASK_DIR" "$CHILD_DIR"
```

## PRD 目标结构（最终）

`prd.md` 应该收敛到：

```markdown
# <任务标题>

## 目标

<为什么 + 做什么>

## 需求

* ...

## 验收标准

* [ ] ...

## 完成定义

* ...

## 技术方法

<关键设计 + 决策>

## 决策（ADR-lite）

背景 / 决策 / 后果

## 范围之外

* ...

## 技术笔记

<约束、引用、文件、研究笔记>
```

---

## 反模式（必须避免）

* 向用户询问可以从代码库推导的代码/上下文
* 在提出具体选项之前让用户选择方法
* 关于是否要研究的元问题
* 狭隘地停留在初始请求上而不考虑演进/边缘
* 让头脑风暴漂移而不更新 PRD

---

## 与开始工作流的集成

头脑风暴完成后（步骤 8 确认批准），流程继续到任务工作流的**阶段 2：准备实施**：

```
头脑风暴
  步骤 0：创建任务目录 + 填充 PRD
  步骤 1-7：发现需求、研究、收敛
  步骤 8：最终确认 → 用户批准
  ↓
任务工作流阶段 2（准备实施）
  代码-规范深度检查（如适用）
  → 研究代码库（基于已确认的 PRD）
  → 配置代码-规范上下文（jsonl 文件）
  → 激活任务
  ↓
任务工作流阶段 3（执行）
  实施 → 检查 → 完成
```

任务目录和 PRD 已经从头脑风暴存在，所以任务工作流的阶段 1 完全跳过。
