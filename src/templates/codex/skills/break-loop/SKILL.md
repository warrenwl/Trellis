# 打破循环 - 深度 Bug 分析

调试完成后，使用此技能进行深度分析，打破"修复 bug -> 忘记 -> 重复"的循环。

---

## 分析框架

从以下 5 个维度分析你刚刚修复的 bug：

### 1. 根因类别

这个 bug 属于哪个类别？

| 类别 | 特征 | 示例 |
|----------|-----------------|---------|
| **A. 缺失规范** | 没有如何做的文档 | 新功能没有检查清单 |
| **B. 跨层契约** | 层之间接口不清晰 | API 返回格式与预期不同 |
| **C. 变更传播失败** | 改了一处，漏了其他 | 更改了函数签名，遗漏了调用点 |
| **D. 测试覆盖缺口** | 单元测试通过，集成失败 | 单独工作，组合后失败 |
| **E. 隐式假设** | 代码依赖未文档化的假设 | 时间戳秒 vs 毫秒 |

### 2. 修复失败的原因（如果适用）

如果尝试了多次修复才成功，分析每次失败：

- **表面修复**：修复了症状，没修复根因
- **范围不完整**：找到了根因，但没覆盖所有情况
- **工具限制**：grep 遗漏了它，类型检查不够严格
- **心智模型**：一直在同一层查找，没想过跨层

### 3. 预防机制

什么机制可以防止这种情况再次发生？

| 类型 | 描述 | 示例 |
|------|-------------|---------|
| **文档** | 写下来让大家知道 | 更新思维指南 |
| **架构** | 从结构上使错误不可能 | 类型安全包装器 |
| **编译时** | TypeScript 严格模式，无 any | 签名更改导致编译错误 |
| **运行时** | 监控、警报、扫描 | 检测孤立实体 |
| **测试覆盖** | 端到端测试、集成测试 | 验证完整流程 |
| **代码审查** | 清单、PR 模板 | "你检查了 X 吗？" |

### 4. 系统性扩展

这个 bug 揭示了什么更广泛的问题？

- **类似问题**：这个问题可能还存在于哪些其他地方？
- **设计缺陷**：是否存在基本的架构问题？
- **流程缺陷**：是否有开发流程改进？
- **知识差距**：团队是否缺少一些理解？

### 5. 知识捕获

将见解固化为系统：

- [ ] 更新 `.trellis/spec/guides/` 思维指南
- [ ] 更新 `.trellis/spec/backend/` 或 `frontend/` 文档
- [ ] 创建问题记录（如果适用）
- [ ] 创建根修复的功能工单
- [ ] 如需要更新检查技能

## 核心哲学

> **调试的价值不在于修复这个 bug，而在于让这类 bug 永远不再发生。**

三个层次的见解：
1. **战术**：如何修复这个 bug
2. **战略**：如何防止这一类 bug
3. **哲学**：如何扩展思维模式

30 分钟的分析节省 30 小时的未来调试。

---

## 分析后：立即行动

**重要**：完成上述分析后，你必须立即：

1. **更新规范/指南** - 不要只列出 TODO，实际更新相关文件：
   - 如果是跨平台问题 → 更新 `cross-platform-thinking-guide.md`
   - 如果是跨层问题 → 更新 `cross-layer-thinking-guide.md`
   - 如果是代码重用问题 → 更新 `code-reuse-thinking-guide.md`
   - 如果是领域特定 → 更新 `backend/*.md` 或 `frontend/*.md`

2. **同步模板** - 更新 `.trellis/spec/` 后，同步到 `src/templates/markdown/spec/`

3. **提交规范更新** - 这是主要输出，而不仅仅是分析文本

> **如果分析停留在聊天中，它就一文不值。价值在于更新的规范。**
