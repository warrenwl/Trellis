# 更新代码规范 - 捕获可执行的契约

当你从调试、实施或讨论中学到有价值的东西时，使用此技能更新相关的代码规范文档。

**时机**：完成任务、修复 bug 或发现新模式之后

---

## 代码规范优先规则（关键）

在这个项目中，实施工作的"规范"意味着**代码规范**：
- 可执行的契约（不仅仅是原则性文本）
- 具体的签名、有效载荷字段、环境密钥和边界行为
- 可测试的验证/错误行为

如果更改涉及基础设施或跨层契约，代码规范深度是必需的。

基础设施/跨层规范的必需部分：
1. 范围 / 触发器
2. 签名（命令/API/数据库）
3. 契约（请求/响应/环境）
4. 验证和错误矩阵
5. 好/基础/坏情况
6. 所需测试（带断言点）
7. 错误 vs 正确（至少一对）

---

## 何时更新代码规范

| 触发器 | 示例 | 目标规范 |
|---------|---------|-------------|
| **实施了功能** | 添加了使用 giget 的模板下载 | 相关的 `backend/` 或 `frontend/` 文件 |
| **做了设计决策** | 使用类型字段 + 映射表实现可扩展性 | 相关代码规范 + "设计决策" 部分 |
| **修复了 bug** | 发现错误处理的微妙问题 | `backend/error-handling.md` |
| **发现了模式** | 找到更好的代码结构方式 | 相关的 `backend/` 或 `frontend/` 文件 |
| **遇到陷阱** | 了解到 X 必须在 Y 之前做 | 相关代码规范 + "常见错误" 部分 |
| **建立了约定** | 团队同意命名模式 | `quality-guidelines.md` |
| **新的思维触发器** | "在做 Y 之前别忘了检查 X" | `guides/*.md`（作为检查清单项目，而非详细规则） |

**关键见解**：代码规范更新不仅仅是针对问题。每个功能实施都包含未来 AI/开发者需要安全执行的设计决策和契约。

## 规范结构概述

```
.trellis/spec/
├── backend/           # 后端编码标准
│   ├── index.md       # 概述和链接
│   └── *.md           # 主题特定规范
├── frontend/          # 前端编码标准
│   ├── index.md       # 概述和链接
│   └── *.md           # 主题特定规范
└── guides/            # 思维检查清单（不是编码规范！）
    ├── index.md       # 指南索引
    └── *.md           # 主题特定指南
```

### 关键：代码规范 vs 指南 - 了解区别

| 类型 | 位置 | 目的 | 内容风格 |
|------|----------|---------|---------------|
| **代码规范** | `backend/*.md`, `frontend/*.md` | 告诉 AI "如何安全实施" | 签名、契约、矩阵、案例、测试点 |
| **指南** | `guides/*.md` | 帮助 AI "考虑什么" | 检查清单、问题、指向规范的指针 |

**决策规则**：问自己：

- 这是**如何编写**代码 → 放入 `backend/` 或 `frontend/`
- 这是**在编写之前**考虑什么 → 放入 `guides/`

**示例**：

| 学习内容 | 错误位置 | 正确位置 |
|----------|----------------|------------------|
| "Windows stdout 使用 `reconfigure()` 而不是 `TextIOWrapper`" | ❌ `guides/cross-platform-thinking-guide.md` | ✅ `backend/script-conventions.md` |
| "编写跨平台代码时记得检查编码" | ❌ `backend/script-conventions.md` | ✅ `guides/cross-platform-thinking-guide.md` |

**指南应该是指向规范的简短检查清单**，而不是复制详细规则。

## 更新过程

### 步骤 1：确定你学到了什么

回答这些问题：

1. **你学到了什么？**（具体点）
2. **为什么重要？**（它防止什么问题？）
3. **它属于哪里？**（哪个规范文件？）

### 步骤 2：分类更新类型

| 类型 | 描述 | 操作 |
|------|-------------|--------|
| **设计决策** | 为什么我们选择方法 X 而不是 Y | 添加到"设计决策"部分 |
| **项目约定** | 我们在这个项目中如何做 X | 在相关部分添加示例 |
| **新模式** | 发现的可重用方法 | 添加到"模式"部分 |
| **禁止模式** | 导致问题的某些东西 | 添加到"反模式"或"不要做"部分 |
| **常见错误** | 容易犯的错误 | 添加到"常见错误"部分 |
| **约定** | 约定的标准 | 在相关部分添加 |
| **陷阱** | 非显而易见的行为 | 添加警告提示 |

### 步骤 3：阅读目标代码规范

在编辑之前，阅读当前的代码规范以：
- 理解现有结构
- 避免重复内容
- 找到更新的正确部分

```bash
cat .trellis/spec/<category>/<file>.md
```

### 步骤 4：执行更新

遵循这些原则：

1. **具体**：包含具体示例，而不仅仅是抽象规则
2. **解释原因**：说明这防止什么问题
3. **展示契约**：添加签名、有效载荷字段和错误行为
4. **展示代码**：为关键模式添加代码片段
5. **保持简短**：每个部分一个概念

### 步骤 5：更新索引（如果需要）

如果添加了新部分或代码规范状态更改，更新类别的 `index.md`。

## 更新模板

### 基础设施/跨层工作的强制模板

```markdown
## 场景：<名称>

### 1. 范围 / 触发器
- 触发：<为什么这需要代码规范深度>

### 2. 签名
### 3. 契约
### 4. 验证和错误矩阵
### 5. 好/基础/坏情况
### 6. 所需测试
### 7. 错误 vs 正确
#### 错误
...
#### 正确
...
```

### 添加设计决策

```markdown
### 设计决策：[决策名称]

**背景**：我们要解决什么问题？

**考虑的选项**：
1. 选项 A - 简要描述
2. 选项 B - 简要描述

**决策**：我们选择选项 X 因为...

**示例**：
\`\`\`typescript
// 如何实现
代码示例
\`\`\

**可扩展性**：将来如何扩展...
```

### 添加项目约定

```markdown
### 约定：[约定名称]

**什么**：约定的简要描述。

**为什么**：我们为什么在这个项目中这样做。

**示例**：
\`\`\`typescript
// 如何遵循这个约定
代码示例
\`\`\

**相关**：相关约定或规范的链接。
```

### 添加新模式

```markdown
### 模式名称

**问题**：这解决什么问题？

**解决方案**：方法的简要描述。

**示例**：
\`\`\`
// 好
代码示例

// 坏
代码示例
\`\`\

**为什么**：解释为什么这更好。
```

### 添加禁止模式

```markdown
### 不要做：模式名称

**问题**：
\`\`\`
// 不要这样做
糟糕的代码示例
\`\`\

**为什么不好**：解释这个问题。

**改为**：
\`\`\`
// 这样做
好的代码示例
\`\`\
```

### 添加常见错误

```markdown
### 常见错误：描述

**症状**：出了什么问题

**原因**：为什么会发生

**修复**：如何纠正

**预防**：将来如何避免
```

### 添加陷阱

```markdown
> **警告**：非显而易见行为的简要描述。
>
> 何时发生以及如何处理。
```

## 交互模式

如果你不确定要更新什么，回答这些提示：

1. **你刚刚完成了什么？**
   - [ ] 修复了 bug
   - [ ] 实施了功能
   - [ ] 重构了代码
   - [ ] 讨论了方法

2. **你学到了或决定了什么？**
   - 设计决策（为什么 X 而不是 Y）
   - 项目约定（我们如何做 X）
   - 非显而易见的行为（陷阱）
   - 更好的方法（模式）

3. **未来的 AI/开发者需要知道吗？**
   - 了解代码如何工作 → 是，更新规范
   - 维护或扩展功能 → 是，更新规范
   - 避免重复犯错 → 是，更新规范
   - 纯一次性实施细节 → 也许跳过

4. **它涉及哪个领域？**
   - [ ] 后端代码
   - [ ] 前端代码
   - [ ] 跨层数据流
   - [ ] 代码组织/重用
   - [ ] 质量/测试

## 质量检查清单

完成代码规范更新之前：

- [ ] 内容具体且可操作？
- [ ] 包含代码示例？
- [ ] 解释了为什么，而不仅仅是做什么？
- [ ] 包含可执行的签名/契约？
- [ ] 包含验证和错误矩阵？
- [ ] 包含好/基础/坏情况？
- [ ] 包含带断言点的所需测试？
- [ ] 在正确的代码规范文件中？
- [ ] 重复了现有内容？
- [ ] 新的团队成员能理解吗？

## 与其他命令的关系

```
开发流程：
  学到东西 → $update-spec → 知识捕获
       ↑                                  ↓
  $break-loop ←──────────────────── 未来会话受益
  （深度 bug 分析）
```

- `$break-loop` - 深度分析 bug，经常揭示需要的规范更新
- `$update-spec` - 实际执行更新（此技能）
- `$finish-work` - 提醒你检查规范是否需要更新

---

## 核心哲学

> **代码规范是活文档。每次调试会话、每个"啊哈时刻"都是使实施契约更清晰的机会。**

目标是**制度性记忆**：
- 一个人学到的东西，每个人都受益
- AI 在一个会话中学到的东西，持久到未来会话
- 错误变成文档化的护栏
