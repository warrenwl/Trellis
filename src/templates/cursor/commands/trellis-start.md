# 开始会话

初始化您的 AI 开发会话并开始处理任务。

---

## 操作类型

本文档中的操作分类如下：

| 标记 | 含义 | 执行者 |
|--------|---------|----------|
| `[AI]` | 由 AI 执行的 Bash 脚本或文件读取 | 您（AI） |
| `[USER]` | 由用户执行的斜杠命令 | 用户 |

---

## 初始化

### 步骤 1：了解 Trellis 工作流 `[AI]`

首先阅读工作流指南以了解开发流程：

```bash
cat .trellis/workflow.md  # 开发流程、约定和快速入门指南
```

### 步骤 2：获取当前状态 `[AI]`

```bash
python3 ./.trellis/scripts/get_context.py
```

返回：
- 开发人员身份
- Git 状态（分支、未提交的变更）
- 最近的提交
- 活动任务
- 日志文件状态

### 步骤 3：读取项目代码规范索引 `[AI]`

根据即将进行的任务，阅读适当的代码规范文档：

**对于前端工作**：
```bash
cat .trellis/spec/frontend/index.md
```

**对于后端工作**：
```bash
cat .trellis/spec/backend/index.md
```

**对于跨层功能**：
```bash
cat .trellis/spec/guides/index.md
cat .trellis/spec/guides/cross-layer-thinking-guide.md
```

### 步骤 4：检查活动任务 `[AI]`

```bash
python3 ./.trellis/scripts/task.py list
```

如果继续之前的工作，请查看任务文件。

### 步骤 5：报告就绪状态并请求任务

输出摘要：

```markdown
## 会话已初始化

| 项目 | 状态 |
|------|--------|
| 开发人员 | {name} |
| 分支 | {branch} |
| 未提交 | {count} 文件 |
| 日志 | {file} ({lines}/2000 行) |
| 活动任务 | {count} |

准备接受您的任务。您想做什么？
```

## 任务分类

当用户描述任务时，对其进行分类：

| 类型 | 标准 | 工作流 |
|------|----------|----------|
| **问题** | 用户询问代码、架构或某事如何工作 | 直接回答 |
| **简单修复** | 拼写错误、注释更新、单行变更、< 5 分钟 | 直接编辑 |
| **简单任务** | 目标清晰，1-2 个文件，范围明确 | 快速确认 → 任务工作流 |
| **复杂任务** | 目标模糊，多个文件，架构决策 | **头脑风暴 → 任务工作流** |

### 决策规则

> **如果有疑问，使用头脑风暴 + 任务工作流。**
>
> 任务工作流确保代码规范被注入到正确的上下文，从而产生更高质量的代码。
> 开销很小，但收益显著。

> **子任务分解**：如果头脑风暴揭示多个独立工作项，考虑使用 `--parent` 标志或 `add-subtask` 命令创建子任务。
> 详情请参阅 `/trellis:brainstorm` 步骤 8。

---

## 问题 / 简单修复

对于问题或简单修复，直接工作：

1. 回答问题或进行修复
2. 如果代码被修改，提醒用户运行 `/trellis-finish-work`

---

## 简单任务

对于简单、定义良好的任务：

1. 快速确认："我理解您想要 [目标]。准备好继续吗？"
2. 如果是，进入**任务工作流阶段 1 路径 B**（创建任务，写 PRD，然后研究）
3. 如果不是，再次澄清并确认

---

## 复杂任务 - 先头脑风暴

对于复杂或模糊的任务，在实现之前先使用 `/trellis-brainstorm` 澄清需求。

摘要：

1. **确认并分类** - 说出您的理解
2. **创建任务目录** - 在 `prd.md` 中跟踪演变的需求
3. **一次问一个问题** - 每次回答后更新 PRD
4. **提出方法** - 对于架构决策
5. **确认最终需求** - 获得明确批准
6. **进入任务工作流** - 在 PRD 中有明确需求

---

## 任务工作流（开发任务）

**为什么使用这个工作流**：
- 在编码前运行专门的研究阶段
- 在 jsonl 上下文文件中配置规范
- 使用注入的上下文实现
- 用单独的检查阶段验证
- 结果：自动遵循项目约定的代码

### 概述：两个入口点

```
从头脑风暴（复杂任务）：
  PRD 确认 → 研究 → 配置上下文 → 激活 → 实现 → 检查 → 完成

从简单任务：
  确认 → 创建任务 → 写 PRD → 研究 → 配置上下文 → 激活 → 实现 → 检查 → 完成
```

**关键原则：研究发生在需求明确之后（PRD 存在）。**

---

### 阶段 1：建立需求

#### 路径 A：从头脑风暴（跳过到阶段 2）

PRD 和任务目录已从头脑风暴存在。直接跳到阶段 2。

#### 路径 B：从简单任务

**步骤 1：确认理解** `[AI]`

快速确认：
- 目标是什么？
- 什么类型的开发？（前端 / 后端 / 全栈）
- 任何特定要求或约束？

如果不清晰，提出澄清问题。

**步骤 2：创建任务目录** `[AI]`

```bash
TASK_DIR=$(python3 ./.trellis/scripts/task.py create "<title>" --slug <name>)
```

**步骤 3：写 PRD** `[AI]`

在任务目录中创建 `prd.md`：

```markdown
# <任务标题>

## 目标
<我们要实现什么>

## 需求
- <需求 1>
- <需求 2>

## 验收标准
- [ ] <标准 1>
- [ ] <标准 2>

## 技术笔记
<任何技术决策或约束>
```

---

### 阶段 2：为实现做准备（共享）

> 两条路径在此汇合。PRD 和任务目录必须存在才能继续。

**步骤 4：代码规范深度检查** `[AI]`

如果任务涉及基础设施或跨层契约，在代码规范深度定义之前不要开始实现。

当变更包含以下任何一项时触发此要求：
- 新增或更改的命令/API 签名
- 数据库模式或迁移更改
- 基础设施集成（存储、队列、缓存、密钥、环境契约）
- 跨层负载转换

继续之前必须有：
- [ ] 确定要更新的目标代码规范文件
- [ ] 定义具体契约（签名、字段、环境密钥）
- [ ] 定义验证和错误矩阵
- [ ] 定义至少一个 Good/Base/Bad 案例

**步骤 5：研究代码库** `[AI]`

根据确认的 PRD，运行专门的研究阶段并产生：

1. `.trellis/spec/` 中的相关规范文件
2. 要遵循的现有代码模式（2-3 个示例）
3. 可能需要修改的文件

使用此输出格式：

```markdown
## 相关规范
- <路径>：<为什么相关>

## 发现的代码模式
- <模式>：<示例文件路径>

## 要修改的文件
- <路径>：<什么变更>
```

**步骤 6：配置上下文** `[AI]`

初始化默认上下文：

```bash
python3 ./.trellis/scripts/task.py init-context "$TASK_DIR" <type>
# type: backend | frontend | fullstack
```

添加研究阶段发现的规范：

```bash
# 对于每个相关规范和代码模式：
python3 ./.trellis/scripts/task.py add-context "$TASK_DIR" implement "<path>" "<reason>"
python3 ./.trellis/scripts/task.py add-context "$TASK_DIR" check "<path>" "<reason>"
```

**步骤 7：激活任务** `[AI]`

```bash
python3 ./.trellis/scripts/task.py start "$TASK_DIR"
```

这设置 `.current-task` 以便钩子可以注入上下文。

---

### 阶段 3：执行（共享）

**步骤 8：实现** `[AI]`

实现 `prd.md` 中描述的任务。

- 遵循注入到实现上下文的所有规范
- 保持变更范围在需求内
- 在完成前运行 lint 和类型检查

**步骤 9：检查质量** `[AI]`

对检查上下文运行质量检查：

- 审查所有代码变更与规范
- 直接修复问题
- 确保 lint 和类型检查通过

**步骤 10：完成** `[AI]`

1. 验证 lint 和类型检查通过
2. 报告实现了什么
3. 提醒用户：
   - 测试变更
   - 准备好时提交
   - 运行 `/trellis-record-session` 记录此会话

## 平台检测

Trellis 根据配置目录自动检测您的平台。对于 Cursor 用户，确保检测正常工作：

| 条件 | 检测到的平台 |
|-----------|-------------------|
| 仅存在 `.cursor/` | `cursor` ✅ |
| 同时存在 `.cursor/` 和 `.claude/` | `claude`（默认）|

如果自动检测失败，手动设置：

```bash
export TRELLIS_PLATFORM=cursor
```

或者在命令前加前缀：

```bash
TRELLIS_PLATFORM=cursor python3 ./.trellis/scripts/task.py list
```
