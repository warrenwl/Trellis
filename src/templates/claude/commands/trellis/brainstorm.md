# Brainstorm - 需求发现（AI 编码增强）

在**实施之前**引导 AI 进行协作式需求发现，针对 AI 编码工作流进行了优化：

* **任务优先**（立即捕捉想法）
* **先行动后提问**（减少低价值问题）
* **技术选择先研究**（避免让用户发明选项）
* **发散 → 收敛**（扩展思维，然后锁定 MVP）

---

## 使用时机

当用户描述开发任务时从 `/trellis:start` 触发，特别是当：

* 需求不清晰或不断演变
* 有多个有效的实施路径
* 权衡很重要（用户体验、可靠性、可维护性、成本、性能）
* 用户可能一开始不知道最佳选项

---

## 核心原则（不可协商）

1. **任务优先（尽早捕捉）**
   确保一开始就有任务，这样用户的想法会被立即记录下来。

2. **先行动后提问**
   如果您可以从仓库代码、文档、配置、规范或快速研究中得出答案——先做那个。

3. **每次只问一个问题**
   不要用一系列问题淹没用户。问一个，更新 PRD，再重复。

4. **倾向于具体选项**
   对于偏好/决策问题，提出 2-3 个可行且具体的方法并说明权衡。

5. **技术选择先研究**
   如果决定取决于行业惯例/类似工具/既定模式，先研究，然后提出选项。

6. **发散 → 收敛**
   在初步理解后，主动考虑未来的演进、相关场景和失败/边缘情况——然后收敛到一个有明确范围之外的 MVP。

7. **不要元问题**
   不要问"我应该搜索吗？"或"你能粘贴代码让我继续吗？"
   如果需要信息：搜索/检查。如果被阻塞：问最小的阻塞问题。

---

## 步骤 0: 确保任务存在（始终执行）

在任何问答之前，确保任务存在。如果不存在，立即创建一个。

* 使用从用户消息派生的**临时工作标题**。
* 标题不完美没关系——稍后在 PRD 中完善。

```bash
TASK_DIR=$(python3 ./.trellis/scripts/task.py create "brainstorm: <short goal>" --slug <auto>)
```

立即用您已知的内容创建/填充 `prd.md`：

```markdown
# brainstorm: <short goal>

## 目标

<一段话：什么 + 为什么>

## 我已经知道的

* <来自用户消息的事实>
* <从仓库/文档中发现的事实>

## 假设（临时）

* <需要验证的假设>

## 开放问题

* <仅阻塞/偏好问题；保持列表简短>

## 需求（演变中）

* <从已知的内容开始>

## 验收标准（演变中）

* [ ] <可测试的标准>

## 完成定义（团队质量线）

* 添加/更新测试（适当的单元/集成测试）
* Lint / typecheck / CI 绿色
* 如果行为改变则更新文档/笔记
* 如果有风险则考虑发布/回滚

## 范围之外（明确说明）

* <我们在此任务中不会做的事情>

## 技术说明

* <检查的文件、约束、链接、引用>
* <研究笔记摘要（如果适用）>
```

---

## 步骤 1: 自动上下文（提问前执行此操作）

在问"代码是什么样子的？"这样的问题之前，自己收集上下文：

### 仓库检查清单

* 识别可能受影响的模块/文件
* 找到现有模式（类似功能、规范、错误处理风格）
* 检查配置、脚本、现有命令定义
* 注意任何约束（运行时、依赖策略、构建工具）

### 文档检查清单

* 查找现有的 PRD/规范/模板
* 查找命令使用示例、README、ADR（如果有）

将发现写入 PRD：

* 添加到"我已经知道的"
* 将约束/链接添加到"技术说明"

---

## 步骤 2: 分类复杂度（仍有帮助，但不阻止任务创建）

| 复杂度   | 标准                                               | 操作                                      |
| ------------ | ------------------------------------------------------ | ------------------------------------------- |
| **简单**  | 单行修复、拼写错误、明显的更改                        | 跳过头脑风暴，直接实施                        |
| **一般**  | 目标清晰，1-2 个文件，范围明确定义                    | 问 1 个确认问题，然后实施                    |
| **中等**  | 多个文件，有些模糊                                   | 轻度头脑风暴（2-3 个高价值问题）              |
| **复杂**  | 目标模糊、架构选择、多种方法                          | 完整头脑风暴                               |

> 注意：任务已从步骤 0 存在。分类只影响头脑风暴的深度。

---

## 步骤 3: 问题门槛（只问高价值问题）

在问**任何**问题之前，运行以下门槛：

### 门槛 A — 我能在没有用户的情况下推导出来吗？

如果答案可以通过以下方式获得：

* 仓库检查（代码/配置）
* 文档/规范/惯例
* 快速市场/开源研究

→ **不要问。**获取它，总结，更新 PRD。

### 门槛 B — 这是元/懒惰问题吗？

示例：

* "我应该搜索吗？"
* "你能粘贴代码让我继续吗？"
* "代码是什么样子的？"（当仓库可用时）

→ **不要问。**采取行动。

### 门槛 C — 这是什么类型的问题？

* **阻塞**：没有用户输入无法继续
* **偏好**：多个有效选择，取决于产品/用户体验/风险偏好
* **可推导**：应该通过检查/研究来回答

→ 只问**阻塞**或**偏好**问题。

---

## 步骤 4: 先研究模式（技术选择的强制要求）

### 触发条件（任一 → 先研究）

* 任务涉及选择方法、库、协议、框架、模板系统、插件机制或 CLI UX 约定
* 用户要求"最佳实践"、"别人怎么做"、"推荐"
* 用户无法合理列举选项

### 研究步骤

1. 识别 2-4 个可比较的工具/模式
2. 总结常见惯例及其存在原因
3. 将惯例映射到我们仓库的约束
4. 为我们的项目产生**2-3 个可行方法**

### 研究输出格式（PRD）

在 PRD 中添加一个部分（要么在技术说明中，要么单独）：

```markdown
## 研究笔记

### 类似工具的做法

* ...
* ...

### 我们仓库/项目的约束

* ...

### 这里可行的方法

**方法 A：<名称>**（推荐）

* 工作原理：
* 优点：
* 缺点：

**方法 B：<名称>**

* 工作原理：
* 优点：
* 缺点：

**方法 C：<名称>**（可选）

* ...
```

然后问**一个**偏好问题：

* "您更喜欢哪种方法：A / B / C（或其他）？"

---

## 步骤 5: 扩展扫描（发散）— 初步理解后必需

在您能总结目标之后，在收敛之前主动扩展思维。

### 扩展类别（每类保持 1-2 点）

1. **未来演进**

   * 这个功能在 1-3 个月内可能变成什么？
   * 哪些扩展点值得现在保留？

2. **相关场景**

   * 哪些相邻命令/流程应该与这个保持一致？
   * 是否有对等期望（创建 vs 更新、导入 vs 导出等）？

3. **失败和边缘情况**

   * 冲突、离线/网络故障、重试、幂等性、兼容性、回滚
   * 输入验证、安全边界、权限检查

### 扩展消息模板（给用户）

```markdown
我理解您想实现：<当前目标>。

在深入设计之前，让我快速发散考虑三类（以避免以后返工）：

1. 未来演进：<1-2 点>
2. 相关场景：<1-2 点>
3. 失败/边缘情况：<1-2 点>

对于这个 MVP，您想包含哪个（或者都不包含）？

1. 仅当前需求（最小可行）
2. 添加 <X>（为未来扩展保留）
3. 添加 <Y>（提高健壮性/一致性）
4. 其他：描述您的偏好
```

然后更新 PRD：

* MVP 中的内容 → `需求`
* 排除的内容 → `范围之外`

---

## 步骤 6: 问答循环（收敛）

### 规则

* 每次消息一个问题
* 尽可能使用多项选择
* 在每个用户回答后：

  * 立即更新 PRD
  * 将回答的项目从`开放问题` → `需求`
  * 用可测试的复选框更新`验收标准`
  * 澄清`范围之外`

### 问题优先级（推荐）

1. **MVP 范围边界**（包含/排除什么）
2. **偏好决定**（提出具体选项后）
3. **失败/边缘行为**（仅对 MVP 关键路径）
4. **成功指标和验收标准**（什么证明它有效）

### 首选问题格式（多项选择）

```markdown
对于 <topic>，您更喜欢哪种方法？

1. **选项 A** — <含义 + 权衡>
2. **选项 B** — <含义 + 权衡>
3. **选项 C** — <含义 + 权衡>
4. **其他** — 描述您的偏好
```

---

## 步骤 7: 提出方法并记录决定（复杂任务）

当需求足够清晰后，提出 2-3 个方法（如果还没通过先研究完成）：

```markdown
根据当前信息，这里有 2-3 个可行的方法：

**方法 A：<名称>**（推荐）

* 如何：
* 优点：
* 缺点：

**方法 B：<名称>**

* 如何：
* 优点：
* 缺点：

您更喜欢哪个方向？
```

在 PRD 中将结果记录为 ADR-lite 部分：

```markdown
## 决定 (ADR-lite)

**上下文**：为什么需要这个决定
**决定**：选择了哪种方法
**后果**：权衡、风险、潜在未来改进
```

---

## 步骤 8: 最终确认 + 实施计划

当开放问题得到解决后，用结构化摘要确认完整需求：

### 最终确认格式

```markdown
以下是我对完整需求的理解：

**目标**：<一句话>

**需求**：

* ...
* ...

**验收标准**：

* [ ] ...
* [ ] ...

**完成定义**：

* ...

**范围之外**：

* ...

**技术方法**：
<简要总结 + 关键决定>

**实施计划（小 PR）**：

* PR1：<脚手架 + 测试 + 最小 plumbing>
* PR2：<核心行为>
* PR3：<边缘情况 + 文档 + 清理>

这看起来正确吗？如果是，我将开始实施。
```

### 子任务分解（复杂任务）

对于具有多个独立工作项的复杂任务，创建子任务：

```bash
# 创建子任务
CHILD1=$(python3 ./.trellis/scripts/task.py create "Child task 1" --slug child1 --parent "$TASK_DIR")
CHILD2=$(python3 ./.trellis/scripts/task.py create "Child task 2" --slug child2 --parent "$TASK_DIR")

# 或链接现有任务
python3 ./.trellis/scripts/task.py add-subtask "$TASK_DIR" "$CHILD_DIR"
```

## PRD 目标结构（最终）

`prd.md` 应收敛到：

```markdown
# <任务标题>

## 目标

<为什么 + 什么>

## 需求

* ...

## 验收标准

* [ ] ...

## 完成定义

* ...

## 技术方法

<关键设计 + 决定>

## 决定 (ADR-lite)

上下文 / 决定 / 后果

## 范围之外

* ...

## 技术说明

<约束、引用、文件、研究笔记>
```

---

## 反模式（坚决避免）

* 询问用户可以从仓库推导出来的代码/上下文
* 在提出具体选项之前让用户选择方法
* 关于是否要研究的元问题
* 狭隘地停留在初始请求而不考虑演进/边缘
* 让头脑风暴漂移而不更新 PRD

---

## 与开始工作流的集成

头脑风暴完成后（步骤 8 确认批准），流程继续到任务工作流的**阶段 2：准备实施**：

```
Brainstorm
  步骤 0：创建任务目录 + 填充 PRD
  步骤 1-7：发现需求、研究、收敛
  步骤 8：最终确认 → 用户批准
  ↓
任务工作流阶段 2（准备实施）
  代码规范深度检查（如适用）
  → 研究代码库（基于确认的 PRD）
  → 配置代码规范上下文（jsonl 文件）
  → 激活任务
  ↓
任务工作流阶段 3（执行）
  实施 → 检查 → 完成
```

任务目录和 PRD 已从头脑风暴中存在，因此任务工作流的阶段 1 完全跳过。
