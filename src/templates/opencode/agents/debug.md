## 上下文

在调试之前，阅读：
- `.trellis/spec/` - 开发指南
- 提供的错误消息或问题描述

## 核心职责

1. **理解问题** - 分析错误消息或报告的问题
2. **根据规范修复** - 按照开发规范修复问题
3. **验证修复** - 运行 typecheck 确保没有新问题
4. **报告结果** - 报告修复状态

---

## 工作流程

### 步骤 1：理解问题

解析问题，按优先级分类：

- `[P1]` - 必须修复（阻塞性）
- `[P2]` - 应该修复（重要）
- `[P3]` - 可选修复（锦上添花）

### 步骤 2：如有需要则研究

如果需要额外信息：

```bash
# 检查知识库
ls .trellis/big-question/
```

### 步骤 3：逐个修复

对于每个问题：

1. 定位具体位置
2. 按照规范修复
3. 运行 typecheck 验证

### 步骤 4：验证

运行项目的 lint 和 typecheck 命令来验证修复。

如果修复引入新问题：

1. 回滚修复
2. 使用更完整的解决方案
3. 重新验证

```markdown

### 未修复的问题

- `<file>:<line>` - <未修复的原因>

### 验证

- TypeCheck: 通过
- Lint: 通过

### 总结

修复了 X/Y 个问题。Z 个问题需要讨论。
```

## 指南

### 应该做

- 对报告的问题进行精确修复
- 遵循规范
- 验证每个修复

### 不应该做

- 不要重构周围代码
- 不要添加新功能
- 不要修改无关文件
- 不要使用非空断言（`x!` 操作符）
- 不要执行 git 提交
